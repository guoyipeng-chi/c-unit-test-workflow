📚 工作流程详解 - 四阶段优化

📍 第一阶段：符号索引缓存 (Symbol Index Caching)
问题： Clang 符号查找会重复扫描编译 Scope，造成性能浪费
解决方案： 一次性构建 Scope 级别的符号索引，使用 _clang_scope_symbol_index_cache 缓存存储
性能提升： 后续查询从 O(n) 线性扫描降低到 O(1) 字典查找
关键代码：
  • _build_scope_symbol_index_with_clang() - 构建符号索引
  • _find_symbol_locations_with_clang() - 查询索引而非重新扫描
  • 缓存键：(mode, scope_files, max_hits_per_symbol)
🔀 第二阶段：问题域智能分流 (Issue Domain Routing)
问题： CMakeLists 错误被错误地路由到测试代码自动修复
解决方案： 使用 LLM 优先分类（温度=0 确保确定性）+ 规则回退
分流规则：
  • 🟦 CMakeLists 问题 → 执行第三阶段（CMake 修复）
  • 🟩 测试代码问题 → LLMTestGenerator 自动修复
  • 🟧 配置禁用时 → 强制返回 test_case 跳过分类
关键代码：
  • _classify_issue_domain() - LLM 优先分类
  • _classify_command_issue_domain() - 规则回退
  • LLM 提示：0 温度，JSON 格式提取
🛠️ 第三阶段：两阶段 CMake 修复 (Two-Stage CMake Repair)
问题： 简单的模板同步无法正确修复复杂的 CMakeLists.txt 配置
解决方案： 分两个阶段执行
第一阶段 - 确定性同步：
  • ensure_cmakelists_for_tests() 生成或更新 CMakeLists 模板
  • 创建 LLM_GENERATED_TEST_SOURCES 块（源文件列表）
  • 创建 LLM_AUTO_TEST_TARGETS 块（foreach 循环生成测试可执行文件）
第二阶段 - LLM 驱动修复：
  • _llm_repair_cmakelists() 读取错误信息和上下文
  • 调用 LLM 生成具体的修复补丁
  • 应用补丁到 CMakeLists.txt 文件
重试机制： 最多 3 次重试，每次失败后重新调用 LLM 获取新的修复方案
⚙️ 第四阶段：配置开关与路由旁路 (Config Control & Routing Bypass)
新增配置参数： cmakelists_autogen_enabled
配置文件位置： llm_workflow_config.json
配置路径： test_generation.compile_fix.cmakelists_autogen_enabled
默认值： false (关闭)
当配置为 true 时：
  • 启用 CMakeLists 自动生成
  • 启用第一和第二阶段的符号索引和智能问题分类
  • 执行第三阶段的两阶段 CMake 修复
当配置为 false 时（推荐）：
  • 跳过所有 CMakeLists 自动生成和修复
  • 旁路问题域智能分流，直接返回 test_case
  • 所有编译/测试失败统一视为测试代码问题
  • 减少不必要的 LLM 调用和分类逻辑
⚡ 性能优化总结
  • ✅ 索引缓存： 符号查询性能提升 100 倍（O(n) → O(1)）
  • ✅ 智能分流： 减少误分类，提高修复成功率
  • ✅ 两阶段修复： 结合确定性同步与 LLM 智能修复，提高 CMake 正确率
  • ✅ 配置控制： 灵活启用/禁用 CMake 自动修复，减少不必要的 API 调用
